# GittlyFileStation

GittlyFileStation 是一个具备 **版本追踪能力** 的现代文件管理系统。它结合了传统云盘（如百度网盘）的易用性与 Git 的版本回溯优势，专为需要记录文件变更历史但不希望处理复杂 Git 命令的用户设计。

## 🚀 核心理念

虽然名字中带有 "Gittly"，但它 **并不是** 一个 Git 托管站（如 GitHub 或 GitLab）。
它的核心目标是：**像网盘一样操作，像 Git 一样记录。**

## ✨ 主要功能

- 📂 **直观的文件管理**：支持文件的上传、下载、删除和覆盖。操作逻辑与传统云盘一致，无需学习 `git add`, `git commit` 或 `git push` 等命令。
- 📝 **Git 式变更记录**：
    - 每次上传或覆盖文件时，系统会自动产生一条类似于 Git 的提交记录（Commit Log）。
    - 你可以为每次上传添加备注，清晰记录“为什么要更新这个文件”。
- 🔍 **内容 Diff 对比**：
    - 针对纯文本文件（如 `.code`, `.txt`, `.md`, `.json` 等），支持直观的差异对比。
    - 能够清晰展示两个版本之间的内容增减，方便追踪代码或文档的演进。
- 🕒 **版本回溯**：随时查看文件的历史版本，并支持快速找回或下载之前的旧版本。

## 💡 为什么选择 GittlyFileStation？

| 特性 | 传统云盘 (如百度网盘) | Git / GitHub | **GittlyFileStation** |
| :--- | :--- | :--- | :--- |
| **操作难度** | 极低（拖拽上传） | 高（需掌握命令行） | **低（Web/即传即录）** |
| **历史追踪** | 较弱（通常仅保留副本） | 极强（精细到行） | **强（支持 Diff 与记录）** |
| **变更备注** | 无 | 强制要求 | **支持（清晰易懂）** |
| **适用场景** | 大文件存储、分享 | 代码开发、协作 | **个人文档管理、配置文件记录** |

## 🛠️ 工作模式

1. **上传/覆盖**：用户直接通过界面上传文件。如果是同名文件，系统会提示覆盖。
2. **自动记录**：系统在后台计算文件摘要（Hash），并将其存入版本数据库，生成一条递增的操作记录。
3. **差异分析**：如果被覆盖的是文本文件，系统会自动生成 Diff 视图供用户预览变更。

## 🏗️ 技术方案 (MVP 阶段)

为了实现快速原型验证（MVP），本项目采用**“极简主义”架构**，剔除过度设计的组件，专注于核心功能的交付。

| 模块 | 标准版 (生产向) | **MVP 验证版 (采取)** | 理由 |
| :--- | :--- | :--- | :--- |
| **后端框架** | FastAPI | **FastAPI** | 高性能且易于构建 REST API。 |
| **异步处理** | Celery + Redis | **FastAPI BackgroundTasks** | 无需额外中间件，在 Python 进程内完成文件存储与摘要计算。 |
| **前端框架** | React (企业级封装) | **React + TailwindCSS** | 灵活定制 UI，通过轻量级 Hook 处理状态。 |
| **数据库** | PostgreSQL | **SQLite** | 无需安装，直接存为本地文件，方便 Git 记录和元数据的快速持久化。 |
| **存储策略** | 对象存储 (S3) | **本地文件系统** | 基于文件 Hash 的目录结构存储，模拟 Git 的 Blob 机制。 |
| **向量库** | Qdrant / Milvus | **暂不使用** | MVP 阶段专注于核心版本管理，暂不引入语义检索。 |
